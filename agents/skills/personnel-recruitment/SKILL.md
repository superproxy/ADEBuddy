# 人员招聘 Skill

把零散的招聘信息整理成可执行的招聘输出，常见产物包括：岗位画像、筛选标准、JD 优化稿、结构化面试题、评分卡、录用建议。

## 何时使用

- 用户要写或优化 `JD`
- 用户要筛选简历、判断候选人匹配度
- 用户要设计技术面试题、评分表、录用建议
- 用户提供的是零散岗位信息，需要先整理成统一招聘口径

## 不要用于

- 猎头话术、员工关系、劳动法判断
- 薪酬测算、offer 审批、背调结论等高风险最终决策
- 证据不足却要求直接下“强推/淘汰”定论的场景

## 输入模板

```markdown
岗位名称：<例如：Java后端工程师（AI Agent + IoT）>

岗位职责：
1. ...
2. ...

任职要求：
1. ...
2. ...

候选人信息（可选）：
- 简历摘要：...
- 工作年限：...
- 项目经历：...
- 技术栈：...
- AI工具使用：...
```

缺字段时不要卡住，先基于现有信息产出首版，并单独列出 `待补充信息`。

## 执行流程

1. **识别任务目标**
   - 先判断用户要的是 `JD优化`、`简历初筛`、`面试题`、`评分卡`、`录用建议` 中的哪几项。
   - 如果用户没说清，就默认输出 `岗位画像 + 筛选标准 + 下一步建议`。

2. **抽取招聘要点**
   - 提炼岗位目标、业务场景、关键技术栈、协作对象、绩效指标。
   - 把要求拆成 `必须具备`、`可培养`、`加分项`，不要混写。

3. **建立评估标准**
   - 给出筛选维度、权重、判定口径和证据要求。
   - 对候选人结论必须写明依据来自哪段经历、项目或技能描述。

4. **生成目标输出**
   - `JD优化`：改写为更清晰的职责、要求、亮点、门槛。
   - `简历初筛`：给出匹配点、缺口、待验证项、是否建议进入下一轮。
   - `面试题`：按基础、项目、系统设计、行为面试分组。
   - `评分卡`：列维度、权重、分值、理由、风险。
   - `录用建议`：输出建议等级、主要风险、建议下一步。

5. **量化并校正结论**
   - 能打分时给出分项分数与总分。
   - 证据不足时把该项写成 `待验证`，不要伪造分数细节。

6. **补行动建议**
   - 根据当前材料，给出下一步动作，例如：补面、补项目细节、验证 AI 工具使用深度、复核级别与薪资区间。

## 🔴 CHECKPOINT

- 用户要求直接做 `录用/淘汰/定级/薪资` 结论，但候选人信息不足时：
  先照常给出初步判断，再明确标注 `该结论不能替代正式面试与审批`。
- 用户提供的岗位要求互相冲突时：
  先指出冲突项，再给出一版可执行口径，不要默默忽略。
- 候选人只有模糊描述、没有项目证据时：
  结论里必须加 `待验证项`，不能写成已确认能力。

## 失败分支与兜底

| 触发条件 | 一线处理 | 仍失败时输出 |
|---|---|---|
| 岗位职责缺失 | 根据岗位名称和已给要求补出基础画像 | 明确列出 `待补充信息`，只输出通用版筛选标准 |
| 任职要求过宽或互相冲突 | 拆成硬门槛、可培养项、加分项 | 标注 `口径待确认`，避免继续细化面试题 |
| 候选人信息太少 | 只做初筛推断，不做强结论 | 输出 `保留判断`，附补充问题清单 |
| 用户只说“帮我招人” | 默认输出岗位画像、筛选标准、面试方向 | 在结尾列出建议补充的岗位信息 |
| 用户要求淘汰/录用但证据不足 | 给风险提示和待验证项 | 明确写 `不建议仅凭当前材料做最终决定` |

## 默认评估模型

适用于 `Java后端 + AI Agent + IoT` 岗位，可按实际岗位调整：

- Java 与框架（SpringBoot / SpringCloud / Spring AI）：`25%`
- 中间件（MySQL / Redis / RabbitMQ）：`15%`
- 云原生（Docker / K8s）：`15%`
- 可观测性（日志 / 链路监控）：`10%`
- 架构与工程能力（高可用 / 扩展性 / 规范 / 敏捷）：`20%`
- AI 编程工具熟练度：`15%`

判定建议：

- `85-100`：强烈推荐
- `70-84`：建议进入下一轮
- `60-69`：保留，需补验证
- `<60`：当前不推荐

## 当前岗位的关键检查点

- 是否真正做过 `AI Agent 平台 / 智能体编排 / 工具调用链路`，而不只是普通 CRUD
- 是否具备 `IoT 设备接入 / 控制链路 / 协议处理 / 稳定性治理` 经验
- 是否能独立完成模块架构设计，并解释技术选型取舍
- 是否有线上 `SLA / 性能 / 稳定性 / 安全` 保障经验
- 是否熟练使用 AI 编程工具，并能给出真实提效案例

## 输出规则

- 先给结论，再给依据，保证读者可以追溯。
- 面试题优先围绕真实项目、线上问题、架构取舍，不要堆纯八股题。
- 对 `必须熟练使用 AI 编程工具` 这样的硬性项，未体现时直接列为缺口或待验证。
- 如果岗位不是 AI Agent + IoT，保留同一输出骨架，只替换评估维度与检查点。

## 反例黑名单

- 不要把 `必须具备` 和 `加分项` 混在一起。
- 不要在证据不足时给出过度确定的录用结论。
- 不要夸大候选人能力，不要替用户脑补项目成果。
- 不要只给总分，不给分项理由。
- 不要输出空泛评价，例如 `综合素质不错`、`技术较全面`，必须具体到经历或能力点。
