# 硬件智能体提示词 Skill

把零散的设备信息、角色设定、产品目标和交互约束，整理为可以直接投喂给模型或工作流平台的提示词成品。

重点是：

- 先查资料，再写提示词
- 优先利用 MCP 的资源、提示模板和工具
- 兼顾设备能力、用户年龄、场景边界和安全限制
- 输出结果可直接复制到平台，必要时保留 `{#InputSlot#}`、`{#LibraryBlock#}` 这类占位语法

## 适用场景

- 用户要为硬件智能体设计系统提示词
- 用户要把某个 IP 角色映射为设备人格
- 用户要为儿童陪伴、提醒、学习、情感陪聊类设备定义话术边界
- 用户要求“先搜索网络资料再生成 prompt”
- 用户要求“尽量用 MCP 的能力”

## 默认输入模板

```markdown
设备名称：<如 AI 闹钟提醒灯>
设备形态：<灯、音箱、机器人、屏幕桌宠、学习机等>
目标用户：<年龄段、身份、典型场景>
角色/IP：<原创角色或现成 IP，如超级飞侠（乐迪）>
核心任务：<提醒、陪聊、天气问答、学习陪伴、习惯养成等>
交互通道：<语音/灯光/屏幕/按键/表情/动作>
环境变量：<时间、地点、记忆、传感器、联网能力>
硬约束：<敏感词、安全要求、长度、语气、禁说内容>
输出平台：<纯 Markdown / 含 InputSlot / 含 LibraryBlock / 其他>
必须搜索项：<IP 资料、硬件规格、儿童安全规范、竞品语气等，可选>
```

用户输入不完整时，不要卡住。先基于已有信息产出首版，并单独列出 `待确认项`。

## 执行流程

### 1. 识别提示词边界

- 明确设备是“纯语音助手”还是“带物理反馈的硬件智能体”。
- 明确角色目标是 `功能优先`、`陪伴优先`、`教育优先` 还是 `品牌/IP 还原优先`。
- 明确是否面向儿童。若是，默认提高安全、简洁、鼓励式表达的权重。
- 明确输出是 `完整系统提示词`、`提示词模板` 还是 `现有模板改写`。

先显式整理这 3 类信息：

- `已确认输入`：用户明确给出的设备事实、角色事实、业务约束
- `缺失输入`：会影响提示词真实性但暂时缺失的信息
- `默认假设`：为了先产出首版而采用的保守假设

如果用户给的是现成模板，先判断它属于哪一类：

1. `可直接改写型`：章节齐全，主要补文案
2. `半成品型`：有部分章节，需要补齐结构
3. `碎片需求型`：只有几条要求，需要先重建骨架

### 1.5 角色事实与设备事实拆分

在动笔前，先把信息拆成两列，避免把 IP 设定误写成设备能力：

| 类别 | 允许写入提示词的内容 |
|---|---|
| 角色事实 | 性格、语气、价值观、关系定位、轻量口头禅 |
| 设备事实 | 可感知内容、可执行能力、不能做的事、平台限制 |

强制规则：

- 角色事实不能覆盖设备事实。
- 超现实能力只能转译成“风格”，不能转译成“真实能力”。
- 如果设备能力未知，禁止脑补屏幕、灯效、定位、摄像头、长期记忆。

### 2. 先做资料检索，优先 MCP

按下面顺序找资料，不要直接闭门写 prompt：

1. 先检查当前环境里是否存在可用的 MCP 工具或文档类 MCP。
2. 若有 MCP 资源、MCP 提示模板、MCP 文档查询能力，优先使用它们读取：
   - 角色/IP 官方设定
   - 产品规格或 PRD
   - 儿童安全、内容审核、品牌语言规范
   - 竞品或同类设备交互风格
3. 若关键信息可能变化，或用户明确要求“搜索网络”，必须联网检索。
4. 当 MCP 无法覆盖时，再使用普通 web 搜索补齐资料。

优先检索这些信息：

- `角色事实`：角色身份、口头禅、价值观、典型语气
- `设备事实`：是否有灯效、屏幕、按键、麦克风、联网、定位、记忆
- `用户事实`：年龄、使用时段、场景、家长预期
- `安全事实`：儿童友好限制、敏感话题边界、误导风险
- `平台事实`：是否支持变量注入、模板占位符、外部记忆块

输出前要简短说明：哪些内容来自资料，哪些内容是基于现有信息做的合理推断。

如需参考 MCP 与提示词设计原则，读取：

- [references/research-playbook.md](references/research-playbook.md)

### 2.5 资料归一化

检索结束后，不要直接写 prompt，先整理成这张表：

| 维度 | 已确认事实 | 来源 | 在提示词中的使用方式 |
|---|---|---|---|
| 角色/IP |  |  | 只映射人格，不映射超能力 |
| 设备能力 |  |  | 只写真实可执行边界 |
| 用户群体 |  |  | 决定词汇难度、互动强度、提醒密度 |
| 场景节奏 |  |  | 决定晨间/日间/夜间策略 |
| 安全边界 |  |  | 决定拒答、降级、风险转介 |
| 平台语法 |  |  | 决定是否保留变量块和条件块 |

### 3. 形成提示词设计决策

先把资料归纳为以下决策，再动笔：

- `角色层`：名字、身份、人格、关系定位
- `任务层`：设备最重要的 3 到 5 个职责
- `感知层`：时间、地点、记忆、传感器、家庭场景怎么进入对话
- `表达层`：语气、句长、儿童友好程度、是否常用昵称
- `边界层`：禁聊、降级、拒答、转移话题、结束语
- `输出层`：是否保留模板占位符、是否需要示例、多语言要求

如果是儿童设备，必须额外明确：

- `依赖感控制`：避免形成“只能依赖这个设备”的表达
- `提醒密度`：避免太吵、太频繁、太强势
- `家长接受度`：避免过度拟人、越界教育、虚假承诺

### 3.5 🔴 CHECKPOINT：高风险决策先停

遇到下面任一情况时，不要默默做决定；先在输出中显式标成 `待确认项`：

- 要把知名 IP 深度还原到儿童设备
- 要增加长期记忆、定位、监护、强安抚等敏感能力
- 要改动用户给出的模板语法，而不是只填充内容
- 角色设定与设备边界或儿童安全存在明显冲突

默认策略：

- 先给 `安全保守版`
- 再列 `确认后可增强项`

### 4. 按固定骨架输出

默认输出 3 个部分：

```markdown
## Part 1. 资料摘要与设计判断
## Part 2. 可直接使用的提示词
## Part 3. 待确认项与可选增强
```

若用户明确只要 prompt，仍建议先给 3 到 6 行资料判断摘要，再给最终提示词。

如果用户提供的是现成模板，输出时遵守这 4 条：

1. 保留原一级、二级标题名称
2. 保留原模板变量块、条件块、库块
3. 不擅自改动块语法、ID、uuid、占位符壳子
4. 只替换块内的自然语言内容，必要时才补新章节

## 提示词结构规则

参考官方提示词写法，优先采用“角色 + 任务 + 约束 + 示例 + 输出格式”的结构，不要把所有要求混成一段散文。

推荐骨架：

```markdown
# 角色设定
## 1. 基本信息
## 2. 性格气质
## 3. 语言表达风格
## 4. 外观与交互设定

# 回复逻辑说明
## 核心任务
## 环境感知
### 时间感知
### 位置感知
### 用户记忆

## 聊天策略
## 失败与降级策略
## 安全边界
## 输出限制
## 示例
```

写作要求：

- 把稳定规则写在前面，把示例放后面
- 把“必须遵守”的要求写成短句或编号，不要埋在长段落里
- 对儿童设备优先使用简短、积极、不诱导依赖的表达
- 若设备输出较短，明确规定单轮长度、句数与废话上限
- 若使用 few-shot 示例，示例只保留 2 到 4 组，避免喧宾夺主

可直接参考骨架与案例：

- [references/prompt-template.md](references/prompt-template.md)

## 生成后自检

生成完成后，按下面清单逐条自检；若任何一项不满足，先修再交付：

- [ ] 是否先给了资料摘要，而不是直接给成品
- [ ] 是否明确区分了“角色人格”和“设备能力”
- [ ] 是否保留了用户原有模板语法
- [ ] 是否没有把 IP 超能力写成设备真实能力
- [ ] 是否给出了时间感知、闲聊收尾、失败降级、安全边界
- [ ] 是否控制了儿童设备的句长、语气和依赖感
- [ ] 是否没有出现“根据情况灵活调整”“可自行发挥”这类空话
- [ ] 是否给出了至少 2 条示例，且示例风格与主规则一致

## 儿童与家庭场景的默认安全规则

除非用户明确要求其他目标，否则面向未成年人的硬件智能体默认加上这些约束：

- 不输出敏感、恐吓、成人化、歧视性内容
- 不鼓励危险行为、夜间沉迷、脱离监护的线下行动
- 不假装拥有现实世界执行能力，除非设备确实具备且用户已授权
- 遇到超出能力范围的问题，要温和降级，不编造事实
- 遇到医疗、法律、财务、极端情绪等高风险主题，保持安抚并建议寻求家长或可信成年人帮助
- 不主动索取隐私信息；如平台已有记忆能力，只能在用户利益明显相关时自然引用

## 输出模板

默认产出采用下面格式：

```markdown
## Part 1. 资料摘要与设计判断
- 已确认：
- 来自检索：
- 推断假设：

## Part 2. 可直接使用的提示词
<在这里给出完整提示词正文>

## Part 3. 待确认项与可选增强
- 待确认项：
- 可选增强：
```

如果用户提供的是现成模板：

- 保留其原有章节名
- 尽量保留其占位符语法
- 只改写内容，不擅自改坏模板语法

## 失败分支与兜底

使用三段式表格，不要只写“有问题就补充”这种空话。

| 触发条件 | 一线处理 | 仍失败时输出 |
|---|---|---|
| 角色/IP 信息不足 | 先写通用人格版，只保留已确认的人格关键词 | 明确要求补官方角色设定或语气样本 |
| 设备能力不明 | 只保留语音提醒和通用陪伴，不写灯效/屏幕/定位等能力 | 在待确认项中列出缺失设备规格 |
| 用户模板语法复杂 | 原样保留变量块、条件块、LibraryBlock，只替换文字内容 | 输出“保守改写版”，说明哪些块未动 |
| MCP 无可用资源 | 改用 web 搜索补角色与安全资料 | 在资料摘要中说明已回退到 web |
| 角色设定与儿童安全冲突 | 保留角色气质，删去不适合儿童设备的表达 | 输出“安全收紧版”，说明删改原因 |
| 用户要求设备具备超现实能力 | 改写为“语言风格模拟”，不改写为真实能力 | 明确说明设备边界，拒绝夸大能力 |

## 针对本 skill 的特殊要求

- 遇到“IP 角色 + 硬件设备”组合时，先区分“角色还原”与“设备可执行约束”，不要只顾像角色。
- 任何需要最新资料、品牌设定、产品规格、法规或竞品信息的请求，都必须先检索，不要仅凭记忆回答。
- 若输出用于闹钟灯、陪伴灯、床头设备，默认强化这些场景：
  - 起床问候
  - 天气播报
  - 上学/作业提醒
  - 夜间安抚
  - 用户主动闲聊后的自然收尾

## 反例黑名单

- 不要把“乐迪会飞、会送快递、会跨国救援”直接写成提醒灯的真实能力。
- 不要为了像 IP 就牺牲设备边界。
- 不要把儿童陪伴写成情感依赖关系。
- 不要篡改用户模板里的 `{#InputSlot#}`、`{#LibraryBlock#}`、`<#if ...>` 语法。
- 不要用“可爱一点”“活泼一点”这种空指令代替可执行约束，必须落到句长、语气、场景和收尾方式。
- 不要只写正向规则，不写失败与降级策略。
- 不要输出冗长样例压过主规则。

## 示例触发语

- “写一个儿童闹钟灯的智能体提示词”
- “参考超级飞侠乐迪，生成提醒灯的角色 prompt”
- “先搜一下这个 IP 和儿童陪伴设备的资料，再给我提示词”
- “把我这段模板改成适合硬件语音智能体的系统提示词”
